# Email Integration — Landing Backend
## Truelite Estates LLP

---

## Context — Why We Built This

When a user submitted a form on any landing page (Gaurs Bento / AU Cosmos),
the lead was saved to Supabase DB but **no notification was triggered**.

The admin had to manually check the Supabase dashboard to know if any lead had arrived.

**Problem:** No real-time awareness — leads could be missed or responded to late.

**Solution:** Every time a new lead is submitted:
- Admin receives an **instant email** with full lead details
- Lead also receives a **confirmation email** (only if they provided their email address)

---

## Technology Choice — Nodemailer + Gmail SMTP

### Options Explored

| Option | Reason Rejected / Accepted |
|--------|---------------------------|
| Resend.com | Best deliverability, supports domain email — but requires new account + domain DNS verification. Better for production. |
| SendGrid | Complex setup, overkill for current scale |
| **Nodemailer + Gmail SMTP** | ✅ **Chosen** — No new account needed, 2-minute setup, free, perfect for testing |

### Why Gmail SMTP

- Uses an existing Gmail account — no new service subscription required
- Gmail App Password used (not actual Gmail password) — secure
- 500 emails/day free — sufficient for current lead volume
- Future migration to `truelitestates.com` domain email requires only changing SMTP credentials in `.env` — no code changes

## Flow — Step by Step

```
1. User fills form on landing page
        ↓
2. Frontend sends fetch() request
   POST /api/enquiry/gaurs-bento  (or au-cosmos)
        ↓
3. Express API receives the request
   enquiryController.js → validates the data
        ↓
4. Lead is saved to Supabase DB
   (Gaurs-Enquries / Au-Enquiries table)
        ↓
5. After successful DB save —
   sendLeadEmails(slug, data) is called
        ↓
6. emailService.js fetches the correct template
   based on slug (gaurs-bento / au-cosmos)
        ↓
7. Nodemailer connects to SMTP server
   (smtp.gmail.com : 587)
        ↓
8. Two emails are sent:

   EMAIL 1 → Admin (always sent)
   ──────────────────────────────────────
   To:      omvir.shishodia@truelitestates.com
   From:    walia3433@gmail.com
   Subject: 🏠 New Lead — Gaurs Codename Bento | 9876543210
   Body:    Name, Phone, Email, Comment, Source (HTML table, branded)

   EMAIL 2 → Lead (only if email was provided)
   ──────────────────────────────────────────────
   To:      lead's email address
   From:    walia3433@gmail.com
   Subject: Thank you for your interest in Gaurs Codename Bento
   Body:    Welcome message + project info + contact number
        ↓
9. Response sent back to frontend — success ✅
   (Email is fired asynchronously — response is never blocked by email)
```

## emailTemplates.js — Why a Separate File

Each landing page has different email content:

- **Gaurs Bento** → Gold (#C8943A) theme, Studio Residences copy, Yamuna Expressway location
- **AU Cosmos** → Dark red (#501313) theme, 3 BHK copy, Siddharth Vihar location

Each project template contains:
- `adminSubject` — Admin email subject (includes phone number for quick reference)
- `adminHtml` — Branded HTML table with full lead details
- `leadSubject` — Lead confirmation email subject
- `leadHtml` — Welcome email with project details and contact number

**To add a new project** — just add a new key in `emailTemplates.js`.
No changes needed in the controller or emailService.

## emailService.js — Key Decisions

### Async Fire & Forget
```js
sendLeadEmails(slug, data).catch(err => console.error('Email send error:', err.message))
```
**Why:** Even if the email fails, the lead's form response should not be blocked.
DB save is the primary operation — email is secondary.
The user should never see an error because of an email failure.

### Conditional Lead Email
```js
if (data.email) {
  // send lead confirmation
}
```
**Why:** Phone is required, email is optional on all forms.
If the lead did not provide an email, skip the confirmation email.
Admin email is always sent regardless.

## .env Configuration

```env
SMTP_HOST=smtp.gmail.com
SMTP_PORT=587
SMTP_USER=walia3433@gmail.com          ← Gmail account used as sender
SMTP_PASS=riknfvcrnbnkpbql             ← Gmail App Password (not actual Gmail password)
ADMIN_EMAIL=omvir.shishodia@truelitestates.com
```

### Why Gmail App Password

Google blocks login via normal password from third-party apps.
App Password = a special 16-digit password generated by Google for a single app only.

Steps taken:
1. Gmail → Security → 2-Step Verification → Enabled
2. App Passwords → App name: TrueliteBackend → Generate
3. 16-digit password received → added to `.env`

## Migrating to Production (truelitestates.com Email)

When professional email credentials are received from the domain provider — change only these 3 lines in `.env`:

```env
SMTP_HOST=mail.truelitestates.com    (or as per provider)
SMTP_USER=enquiry@truelitestates.com
SMTP_PASS=password for that email
```

No code changes required anywhere — everything else stays the same.

## Adding a New Landing Page in Future

Just add a new entry in `emailTemplates.js`:

```js
'new-project-slug': {
  projectName:  'Project Name',
  location:     'City, Area',
  developer:    'Developer Name',
  adminSubject: (data) => `🏠 New Lead — Project Name | ${data.phone}`,
  adminHtml:    (data) => `...branded HTML with lead details...`,
  leadSubject:  ()     => `Thank you for your interest in Project Name`,
  leadHtml:     (data) => `...welcome email HTML...`,
}
```

The controller picks it up automatically via the slug — no other file needs to be touched.
